# Database Migration Strategy

Database migrations for SQLAlchemy projects use Alembic for schema changes, with data migrations handled separately via standalone scripts. This separation avoids coupling and simplifies rollbacks.

## Key Points

- Use Alembic for schema migrations (`alembic revision --autogenerate`)
- Keep data migrations as separate standalone scripts in `scripts/migrations/`
- Always review autogenerated migrations — they can miss renamed columns
- Test migrations against a copy of production data before deploying

## Details

Alembic's autogenerate feature compares your SQLAlchemy models against the current database schema and produces migration scripts. However, it cannot detect column renames (it sees a drop + add instead), so manual review is essential.

Data migrations — backfilling columns, transforming data formats, etc. — should not live inside Alembic migration files. They have different rollback characteristics and testing requirements.
